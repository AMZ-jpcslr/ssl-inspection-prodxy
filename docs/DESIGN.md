# 設計ドキュメント

## アーキテクチャ

本プロトタイプは、従業員端末のブラウザに「明示プロキシ」を設定して利用する想定。

本リポジトリでは、プロキシ本体と管理ダッシュボードを **同一の Node.js プロセス** で起動します。
（`src/main.js` が両方を起動）

```
┌────────────────────┐
│  従業員端末（Browser） │
└─────────┬──────────┘
		  │ HTTP/HTTPS (CONNECT)
		  ▼
┌──────────────────────────────┐
│ 通信検査プロキシ (Node.js)     │
│ - SSL inspection (MITM)        │
│ - ドメインブロック              │
│ - アクセスログ永続化(JSONL)     │
└─────────┬────────────────────┘
		  │
		  │ HTTP/HTTPS
		  ▼
┌────────────────────┐
│ 外部サービス         │
│ (google.com 等)      │
└────────────────────┘

┌──────────────────────────────┐
│ 管理ダッシュボード (Express)  │
│ - ログ一覧表示                 │
│ - ブロック設定編集              │
│ - 監査ログ閲覧                  │
└──────────────────────────────┘
```

### 主なモジュールと責務

- `src/config.js`
	- `config.json` を読み込み、必要なら `config.local.json` で上書きマージします（機密をGit管理外にするため）
	- Docker等を想定し、一部の設定を環境変数で上書きできます
- `src/main.js`
	- MITMプロキシ（HTTP/HTTPS）の起動と、リクエスト/レスポンスのフック
	- ルールに応じたブロック（ドメインブロック）
	- ログの生成（JSONL追記）
- `src/dashboard/server.js`
	- ダッシュボード（一覧/詳細/PII表示/監査ログ/設定更新）
	- （有効時）ログイン/ログアウトとセッションCookie検証
- `src/dashboard/auth.js`
	- scryptパスワードハッシュ検証、署名付きセッションCookie（HMAC）生成/検証、Cookieパース
- `src/dashboard/render.js`
	- HTML文字列レンダリングとXSS対策（escape）
- `src/logStore.js`
	- アクセスログのJSONL追記と末尾N件読み込み
- `src/policyStore.js`
	- ブロック対象ドメインの動的更新（メモリ即時反映 + JSON永続化）
- `src/auditStore.js`
	- 監査ログのJSONL追記と末尾N件読み込み
- `src/httpBody.js` / `src/pii.js`
	- 本文デコード（圧縮/テキスト判定）と、メールアドレス検出（マスク表示）

### 永続化ストレージ（ファイル）

- `data/access.log.jsonl`: アクセスログ（1行=1JSON）
- `data/audit.log.jsonl`: 監査ログ（ログイン/ログアウト/設定変更など）
- `data/policy.json`: 動的ブロックリスト（ダッシュボードから保存される）
- `data/files/`: 画像などをファイル保存する場合の保存先（ログから参照URLで辿れる）
- `.http-mitm-proxy/`: `http-mitm-proxy` が生成する MITM 用のCA/証明書/鍵キャッシュ

### 実行形態（ローカル / Docker）

- ローカル実行: `npm start`
	- Proxy(8080) と Dashboard(3001) を同一プロセスで起動します
- Docker実行: `docker compose up --build`
	- `./data` と `./.http-mitm-proxy` をボリュームで永続化します（ログとMITMキャッシュの保持）

## 技術選定の理由

JavaScript(Node.js)で統一し、プロキシとダッシュボードを同一プロセスで起動できる構成にした。

- `http-mitm-proxy`: HTTPSを復号してリクエスト/レスポンス情報を取得し、検査・ブロック・ログ化しやすい
- `express`: ログ閲覧UIを最小で実装できる
- ログ永続化はJSONL: DBの導入やネイティブ依存を避け、プロトタイプとして運用しやすい
